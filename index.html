üåêDNS - Domain Name System

DNS stands for Domain Name System.
It's like the phonebook of the internet üìñ

We humans remember names like: https://shoesy.xyz

But computers work with IP addresses like: 3.108.235.188

üß† So DNS is the system that converts domain names to IP addresses.

üîç What DNS does behind the scenes:

1Ô∏è‚É£ Browser Request

Browser asks DNS: "Hey, what's the IP address of shoesy.xyz?"

2Ô∏è‚É£ DNS Response

DNS replies: "It's 3.108.235.188."

3Ô∏è‚É£ Connection

Browser connects to that IP to load your Node.js app hosted on AWS.

‚öôÔ∏èPM2 - Process Manager

PM2 is a production process manager for Node.js applications.
It helps you run, monitor, and manage your app easily ‚Äî especially when it's deployed (like on AWS).

üîÑ Automatic Restart on Crash

If your app crashes due to an error, PM2 will automatically restart it. üõ° This keeps your app always running, 24/7.

üîÅ Startup Script (Auto Start on Boot)

After server reboot (like AWS restart), PM2 can auto-start your app. No need to manually run node app.js again.

pm2 startup
pm2 save

üß† Process Monitoring

You can see your app's status: online/offline, memory usage, CPU usage, restarts.

pm2 list
pm2 status

üì¶ Multiple App Management

You can manage many apps at the same time.

pm2 start api.js
pm2 start admin.js
pm2 stop all
pm2 restart all

üìú Log Management

View real-time logs for debugging:

pm2 logs
pm2 logs app
pm2 logs --lines 100

üåê Cluster Mode for Load Balancing

PM2 can run your app in cluster mode, using all CPU cores. This boosts performance in production.

pm2 start app.js -i max

üîÑReverse Proxy & NGINX

A reverse proxy is a server that sits in front of your app and forwards client requests to the correct backend service.
üëâ Think of it as a gatekeeper for your app.

Reverse proxy = middleman between users and backend app.

You must use one (like NGINX) when deploying to production.

It helps with HTTPS, security, performance, and scalability.

üõ† NGINX Alternatives:

‚úÖ Apache HTTP Server - Traditional and powerful
‚úÖ Caddy - Automatic HTTPS
‚úÖ HAProxy - High performance load balancer

üîêSSH - Secure Shell

SSH stands for Secure Shell.
It's a secure way to connect to a remote server (like your AWS EC2) from your local computer using the terminal.

You use SSH when you type something like:

ssh ubuntu@your-ip-address

üîí It encrypts everything ‚Äî so no one can spy on your commands or data.

Term Meaning
SSH Secure way to connect to a server via terminal
Private Key (.pem) Secret file you keep locally
Public Key Key stored on the server to match your private
Authentication Key-based (more secure than password)

üîóSymlinking

Symlinking (short for symbolic linking) is like creating a shortcut to a file or folder.

‚úÖ Simple Definition:
A symlink is a pointer or reference to another file or directory.

It's similar to the shortcut icon you create on your desktop in Windows. The shortcut doesn't contain the actual file ‚Äî it just points to it.

‚òÅÔ∏èAWS EC2 Essentials

Amazon EC2 (Elastic Compute Cloud) provides scalable computing capacity in the cloud.

üñ•Ô∏è Instance Types

üîíHTTP vs HTTPS & SSL

‚ùå HTTP (Port 80)

  • ‚Ä¢ Data sent in plain text
  • ‚Ä¢ No encryption
  • ‚Ä¢ Vulnerable to attacks
  • ‚Ä¢ Not trusted by browsers
  • ‚Ä¢ Poor SEO ranking

‚úÖ HTTPS (Port 443)

  • ‚Ä¢ Data encrypted with SSL/TLS
  • ‚Ä¢ Secure communication
  • ‚Ä¢ Protected from attacks
  • ‚Ä¢ Browser trust & green lock
  • ‚Ä¢ Better SEO ranking

SSL (Secure Sockets Layer) / TLS (Transport Layer Security)
SSL certificates encrypt data between your server and users' browsers. It's essential for any production website.

üÜì Free SSL with Let's Encrypt & Certbot:

# Install Certbot
sudo apt install certbot python3-certbot-nginx -y

# Get SSL certificate for your domain
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

# Auto-renewal (runs twice daily)
sudo systemctl enable certbot.timer
sudo systemctl start certbot.timer

# Test renewal
sudo certbot renew --dry-run

üìã SSL Certificate Types

Domain Validation (DV): Basic, free with Let's Encrypt
Organization Validation (OV): Business verification
Extended Validation (EV): Highest trust, green bar

üîÑ HTTPS Redirect

Always redirect HTTP to HTTPS to ensure security:

# Nginx config

listen 80;
return 301 https://$server_name$request_uri;
}

üõ°Ô∏èSecurity Groups & Firewall

Security Groups act as virtual firewalls for your EC2 instances.
They control inbound and outbound traffic based on rules you define.

üîí Essential Security Group Rules:

Type Port Source Purpose
SSH 22 Your IP only Remote server access
HTTP 80 0.0.0.0/0 (Anywhere) Web traffic (redirect to HTTPS)
HTTPS 443 0.0.0.0/0 (Anywhere) Secure web traffic
Custom 3000 Localhost only Node.js app (behind proxy)

üö® Security Best Practices

Principle of Least Privilege: Only open necessary ports
SSH Key Only: Disable password authentication
Regular Updates: Keep system packages updated
Fail2Ban: Block repeated failed login attempts

üîê UFW (Ubuntu Firewall)

# Enable UFW

# Allow specific ports
sudo ufw allow 22
sudo ufw allow 80
sudo ufw allow 443

# Check status

üåçEnvironment Variables

Environment variables store configuration data outside your code.
They keep sensitive information like API keys, database URLs, and settings secure.

üìÅ .env File Structure

DB_PORT=5432

# API Keys

# App Settings

üîí Security Best Practices

Never commit .env: Add to .gitignore
Use strong secrets: Generate random strings
Separate environments: dev, staging, production
Rotate keys regularly: Update API keys periodically

‚öôÔ∏è Setting Environment Variables

# Method 2: In ~/.bashrc

# Method 3: PM2 ecosystem file

}

üöÄDeployment Strategies

Deployment is the process of moving your code from development to production.
A good deployment strategy minimizes downtime and reduces risks.

üì¶ Manual Deployment

# 1. Connect to server
ssh -i key.pem ubuntu@your-ip

# 2. Navigate to app directory
cd /path/to/your/app

# 3. Pull latest changes
git pull origin main

# 4. Install dependencies
npm install

# 5. Restart application
pm2 restart myapp

üîÑ Zero-Downtime Deployment

# Using PM2 cluster mode
pm2 start app.js -i max

# Graceful reload (zero downtime)
pm2 reload myapp

# Or restart with delay
pm2 restart myapp --update-env

ü§ñ Automated Deployment (CI/CD)

GitHub Actions Example:

# .github/workflows/deploy.yml
name: Deploy to EC2
on:
push:
branches: [ main ]
jobs:
deploy:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v2
- name: Deploy to server
run: |
ssh user@server 'cd app && git pull && pm2 restart myapp'

üîß Deployment Checklist

‚úÖ Environment variables set
‚úÖ SSL certificate configured
‚úÖ Database backed up
‚úÖ Security groups updated
‚úÖ Monitoring alerts active
‚úÖ Domain DNS configured

üîÑ Blue-Green Deployment

Strategy: Two identical production environments
Blue: Current live version
Green: New version being deployed
Switch: Route traffic from Blue to Green
Rollback: Quick switch back if issues occur

üöÄ Docker Deployment

WORKDIR /app

# Build and run

docker run -p 3000:3000 myapp
